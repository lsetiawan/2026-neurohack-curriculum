Lesson 01 · Why pixi?
1:30 – 1:42

Not bad luck — the default outcome

  • Neither conda nor pip writes a lock file unless you go out of your way (pip freeze, conda env export) — so a requirements.txt is a wish, not a record.
  • Named conda environments are shared and long-lived — one pip install in the wrong terminal and the env your other project depends on is different tomorrow.
  • Whether an env is even active is invisible state — it lives in your shell, not the project. Running the wrong Python is trivially easy.
Failure mode 1 — drift Failure mode 2 — non-reproducibility
"Reproducible" — the narrow meaning we'll use all session
Same inputs + same environment
⇒ same outputs, on anyone's machine.

Not "runs for me." Not "worked in March."
Bit-for-bit, on a machine that has never seen your project before.

What pixi does differently

"But uv already does this?"

You don't give uv up — pixi uses it for the PyPI half of the solve:

PyPI

→

  • The difference is scope. uv resolves PyPI only; pixi adds the whole conda-forge world — the Python interpreter itself, plus compiled libraries that don't ship as working wheels (GDAL's C libs, CUDA, MKL).
  • Pixi's own advice: keep deps in the conda [dependencies] table where you can; reach for [pypi-dependencies] only when a package isn't on conda-forge.

Lesson 03 · Tasks & the lock file
📖 display-only — just read

The same trick in CI

.github/workflows/pixi-tutorial-example.ymlcondensed
name: pixi-tutorial-example
on:
  push: # …paths: lsetiawan-pixi/example/**
jobs:
  reproduce:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: prefix-dev/setup-pixi@v0.10.0
        with:
          manifest-path: lsetiawan-pixi/example/pixi.toml
          locked: true
      - run: pixi run analyze
        working-directory: lsetiawan-pixi/example
  • setup-pixi installs pixi itself on the runner.
  • locked: true — install exactly what's in pixi.lock, and fail the job outright if lock and manifest have drifted apart. No silent re-solve on someone else's machine.
  • This exact file keeps lsetiawan-pixi/example/ green in this repo, every time someone touches it.

Lesson 03 · Tasks & the lock file
Debrief

The reproducibility contract

  • Tasks make the project self-documenting — the commands live in pixi.toml, not in your head or a README.
  • The lock file is the contract — exact versions and hashes, per platform. It's the thing you commit.
  • Platforms + CI extend it to machines you don't own — locked: true turns "works on my machine" into an enforced guarantee.

Lesson 04 · Beyond one environment
2:25 – 2:40

The problem with one environment

Your users need…
numpy, matplotlib — enough to run analyze.py. Nothing more.
You also want…
pytest to test it, maybe a linter, maybe an IPython shell to poke at it interactively.
  • If it all lands in one [dependencies] table, everyone — CI, collaborators, a user who just wants the figure — pulls down your entire dev toolchain.
  • Installs get slower, the solve gets harder, and "what does this project need" gets muddied with "what do I need to work on it."
  • Conda's traditional fix — a second environment-dev.yml — drifts the moment someone updates one file and forgets the other.
Lesson 04 · Beyond one environment
2:25 – 2:40

Pixi's answer: features → environments

One manifest, one lock file. Declare features — named groups of extra dependencies — then compose them into environments.

[dependencies]
runtime deps only
→
default env
what users get
feature "dev"
+ pytest
→
dev env
opt-in, for you

The default environment is still just [dependencies]. Everything else is opt-in.

Lesson 04 · Beyond one environment
2:25 – 2:40

A dev environment in two commands

terminalscratch/monte-carlo

Debrief

The default environment is your users' contract

Lesson 05 · For package maintainers
2:40 – 2:53

Analyses are projects; libraries are packages

  • Everything so far treated scratch/monte-carlo as a standalone analysis — it owns its own pixi.toml, and nobody pip installs it.
  • A package you maintain already has a pyproject.toml: pip reads it, uv reads it, your CI reads it — and if you publish, PyPI reads it too.
  • Pixi does not ask you to replace that file. It moves in alongside your metadata, under [tool.pixi.*] — one file, read by every tool in your stack.

Lesson 05 · For package maintainers
2:40 – 2:53

One file, two audiences

terminal~/curriculum
pixi init --format pyproject scratch/pi-lib && cd scratch/pi-lib
cat pyproject.toml
pyproject.toml · 1standard — pip & uv read this
[project]
name = "pi-lib"
version = "0.1.0"
dependencies = []
requires-python = ">= 3.11"

[build-system]
build-backend = "hatchling.build"

requires-python does double duty — pixi derives its python dependency from it. hatchling is the ecosystem-default backend.

pyproject.toml · 2pixi's space — layered on top
[tool.pixi.workspace]
channels = ["conda-forge"]
platforms = ["osx-arm64"]

[tool.pixi.pypi-dependencies]
pi_lib = { path = ".", editable = true }  # ★

[tool.pixi.tasks]

★ the line to notice — your own package, installed editable into every pixi env: importable straight from the scaffolded src/pi_lib/__init__.py. No pip install -e .

Prove the editable install

import numpy as np

def estimate_pi(n_samples, rng=None):
    if rng is None:
        rng = np.random.default_rng(42)
    xy = rng.random((n_samples, 2))
    inside = (xy**2).sum(axis=1) <= 1.0
    return 4 * inside.mean()
tests/test_pi_lib.py
from pi_lib import estimate_pi

def test_estimate_pi_converges():
    assert abs(estimate_pi(50_000) - 3.14159) < 0.05

1 passed in 2.73s

Lesson 05 · For package maintainers
Debrief

One pyproject.toml, every tool

  • [project] stays 100% standard — pip users, uv users, and PyPI publishing are completely untouched by anything pixi adds.
  • [tool.pixi.*] layers on top — environments, tasks, conda overrides, with one lock file covering every environment your contributors need.
  • Extras map to features — pixi run -e test mirrors pip install .[test]. One list to maintain, not two.
